# V1.7 · 多 horizon 期限结构预测 + 二阶/轨迹特征(post-V1.6)

> 定位:V1.6([v1.6-progress.md](v1.6-progress.md),因子层 + 信心校准评估)**已落地**之后的一段
> **预测粒度 + 特征深度**增强。不改「代码算特征 → LLM 只解释」骨架,做两件**互补但独立**的事:
> 1. **多 horizon 预测**:把"每资产只给 AI 自选的一条预测"改成"每资产 × 固定 horizon 网格各给一条",
>    让 AI 被迫表达**短/中/长的期限结构**,同时让评估层的 asset×horizon 网格从稀疏变密。
> 2. **二阶 / 轨迹特征**:在因子层加少量"代码算的"二阶信号(指标变化率/加速度、动量轨迹形状),
>    给 AI 建期限结构的**原料**(当前 prompt 只有多窗口**水平值**,没有二阶项)。
>
> 时间:2026-06-25 起设计。**本文件为设计 + 分步计划(尚未实现,`☐` 待建)**。

---

## 0. 动机

两点观察(讨论记录,2026-06-24/25):

1. **当前每资产只产一条预测**(schema "各给且只给一条",AI 自选 horizon)。出发点是"只在信号最清楚的
   期限表态",但副作用是**评估稀疏不可比** —— scorecard 的 asset×horizon 网格每天只点亮 AI 碰巧选的
   那一格,大片空白,看不出"模型在哪个期限有 skill"。

2. **prompt 里"过往走势"已大体被当日多窗口指标吸收**(ret_20 / px_vs_ma200 / z_252 …),
   但缺**两类信息**:(a) 指标自身的**历史/变化率**(动量在加速还是衰竭 = 二阶);
   (b) **轨迹形状**(同样涨幅,平稳爬 vs 跳空堆出来,后续表现不同 —— 见 frog-in-the-pan)。
   光强制多 horizon 不会自动用上二阶信息;**得把二阶特征喂进 prompt**,两件事一起做才完整。

> 业内对照:多窗口 + 滞后/二阶特征喂 **ML 模型**是标配;预测**收益期限结构**也是常规。
> 但"喂 LLM 直接判方向"本身是探索区,无成熟做法可抄 → 一律走 A/B + scorecard 裁决。

## 1. 多 horizon 期限结构预测

**改动本质 = schema + prompt;下游几乎零改动**(V1.6 已把账本/结算/评估全按 horizon 设计):

| 环节 | 现状 | 多 horizon |
|---|---|---|
| 账本 key | `(date, model, asset, **horizon**, arm)` | ✅ 无需改(horizon 已在 key) |
| 结算 | `HORIZON_DAYS[horizon] → realized_move(…, n)` | ✅ 无需改(按各 horizon 算真实走势) |
| scorecard | `for h in {horizons}` 建 asset×horizon 格 | ✅ 无需改(本就是网格,只是喂得密了) |
| 因子基线 `_factor` | 按 `(date, asset, horizon)` 去重 | ✅ 无需改 |

**设计选择**:
- **网格 = {h_5d, h_20d, h_60d}**(短/中/中长)。**砍掉 next_1d**(单日方向纯噪音 ~50%,留着只稀释)。
- **真·长期(h_120d/252d)暂不上**:结算要等约一年,数据回不来;项目还年轻,先攒短中。
- schema 唯一性约束:从"asset 不重复"→"**asset×horizon 不重复**";每个 (asset,horizon) 各一条
  {direction, confidence, key_factors, if_then, invalidation}。预测量:4 资产 ×3 horizon = 12 条/模型/臂。
- **低把握给低 confidence**(纪律不变):被迫给某期限不代表硬凑方向,校准评分会抓过度自信。

**收益**:scorecard 立刻变密 → 命中率页能看完整期限结构;校准/技能数据积累快 3×。

## 2. 二阶 / 轨迹特征(因子层)

在 `factors.py` 加**少量、代码算的**二阶信号(**不喂原始日线序列** —— LLM 读长数字串易错、撑 token、
诱导自行心算违背分层)。候选(择 2~3 个起步,走 A/B):

- **动量加速度**:`ret_20(今天) − ret_20(20天前)`、或均线斜率变化 —— 描述"动量在加速/熄火"(当前快照没有)。
- **信号轨迹**:因子 composite 在 20/60 天前的值(让 AI 看到信号**轨迹**而非只有水平)。
- **轨迹形状标量(frog-in-the-pan)**:连续同向日占比 / 跳空贡献占比 —— 平稳 vs 跳跃的动量质量。
  参考:Da, Gurun, Warachka (2014);时序动量 Moskowitz, Ooi, Pedersen (2012)。

入 prompt 时同样走"代码算好的数字 + 白话标签",作为因子打分段的扩展。

> 诚实预期:对**短期方向**增量很小(短期近不可测);对**校准**和中期趋势可能更有用。
> 别凭"应该有用"上 —— 当 code feature,前向 scorecard 裁决。

### 实现记录 / 经验(2026-06-25)
- **二阶特征进的是"价格特征块",两臂(A/B)都看得到** —— 所以**新闻 A/B 无法隔离"二阶有没有用"**。
  二阶的价值只能靠 scorecard 的**技能(命中 − 基线)随前向时间**来判,或将来单开一个"含/不含二阶"的消融臂。
  本版先把它当普通 code feature 喂双臂,不为它加第三臂(省一半 LLM 调用)。
- **多 horizon 下游零改的红利兑现**:账本 key `(date,model,asset,horizon,arm)` 本就含 horizon、结算/scorecard 本就按 horizon
  → `predictions.record` / `evaluate` **一行没改**,每模型每臂从 4 条自动变 12 条。**唯一要改的是 render 三处**
  (夹紧 / 共识 / 回填)+ schema/prompt。
- **`fip_60`(frog-in-the-pan)**:`sign(ret_60) × (跌日占比 − 涨日占比)`;<0 = 连续小步动量(更易延续),>0 = 跳空式(易反转)。

## 4. 与 v1.8 的协同(共享时序工具箱)

v1.7 与 [v1.8(新闻指标化)](v1.8-progress.md) **同期开发**,接口在此对齐(总排期见 [协同开发计划](v1.7-v1.8-plan.md)):

- **共享 `tsfeatures` 时序工具箱**:v1.7 的二阶/轨迹特征,与 v1.8 的"时序动态族"(情绪 MA/动量/加速度/z/反转)
  **是同一套数学,只是喂的序列不同**(价格 vs 情绪)。→ v1.7 实现二阶时**把通用算子抽成 `tsfeatures.py`**
  (`rolling_mean / momentum / acceleration / zscore / slope / reversal_flag / streak / dispersion`),v1.8 直接复用。
  **v1.7 先做,顺手建出这个工具箱。**
- **分歧度 = 不确定性 → 校准**(借自 v1.8):可加**跨模型预测分歧 / 跨 horizon 因子分歧**,越不一致越调低信心,直接喂校准。
- **背离 = 上下文**(借自 v1.8):可加**因子—价格背离 / 期限结构背离**(5d 看多但 60d 看空 → 点名矛盾)。
- **"异常 z" 优于"绝对值"**:二阶特征优先用 z 化/标准化的变化量(更可比、省 LLM 心算)。
- **诚实纪律一致**:二阶特征当 code feature 走 A/B + scorecard 裁决;期望对方向增量小、对校准更有用。

## 5. 不做 / 边界

- **不喂原始日线序列**给 LLM(只喂代码算的二阶标量)。
- **不上 h_120d/252d**(结算太慢,数据回不来);等前向积累够再议。
- 二阶特征**不进 A 臂之外的硬基线**之前,先观察其对校准的影响,避免无效复杂度。
- 与 V2 的关系:V2 的回填引擎/区间聚合仍归 V2;本版只动"单日预测的粒度与特征深度"。
